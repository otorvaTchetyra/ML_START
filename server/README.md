## Задача 1: Алгоритм подсчёта гранул и расчёта интенсивности
**Срок:** 16.04 – 21.04  
**Статус:** Выполнено

### Что реализовано

Модуль `models/counter.py` — класс `GranuleCounter`.

Принимает результаты детекции с каждого кадра и возвращает:
- количество гранул в кадре
- интенсивность подачи корма в гранулах/сек
- интенсивность подачи корма в гранулах/мин

**Алгоритм расчёта интенсивности** построен на скользящем окне: накапливаются записи `(время, количество)` за последние N секунд (по умолчанию 5), после чего суммарное число гранул делится на фактическую длительность окна. Такой подход даёт сглаженный показатель без всплесков при единичных кадрах с высокой плотностью.

Дополнительно реализовано:
- сброс состояния между сессиями (`reset()`)
- накопительный счётчик гранул и кадров за сессию
- валидация параметра окна

## Задача 2: Интеграция модели с сервером и тестирование инференса
**Срок:** 22.04 – 25.04  
**Ответственные:** Ковпак Олеся, Tchetyrkin Daniil  
**Статус:** Выполнено

### Что реализовано

Модуль `models/detector.py` — класс `GranuleDetector`.

Обёртка над YOLOv8 (ultralytics): принимает кадр в формате BGR (`numpy.ndarray`), запускает инференс и возвращает список объектов `Detection` с координатами bounding box и значением confidence. Параметры `confidence` и `iou` можно менять на лету без перезапуска сервера.

Модуль `api/stream.py` — интеграционный слой между детектором, счётчиком и FastAPI:

- `POST /stream/upload` — принимает видеофайл (MP4 или AVI), покадрово прогоняет через детектор и счётчик, результаты каждого кадра отдаёт клиенту в реальном времени в формате NDJSON
- `POST /stream/stop` — останавливает обработку
- `GET /stream/status` — текущий статус и накопленная статистика сессии

Детектор и счётчик существуют как единственный экземпляр на весь процесс и инициализируются лениво при первом обращении.

## Поправки: Трекинг гранул, точность подсчёта и настройки обработки
**Дата:** 11.05.2026  
**Статус:** Выполнено

### Проблемы, которые были выявлены

1. **Двойной счёт гранул.** Одна и та же гранула, видимая в нескольких кадрах подряд, засчитывалась в каждом кадре заново. `total_granules` был завышен.
2. **Некорректные timestamps.** Время кадра бралось через `monotonic()` — реальное время CPU, а не время в видео. При медленном инференсе скользящее окно и интенсивность считались неверно.
3. **Избыточная нагрузка.** YOLOv8 запускался на каждом кадре, хотя гранулы между кадрами почти не меняются.
4. **Несовместимость зависимостей.** `passlib 1.7.4` несовместима с `bcrypt 5.x` — сервер не поднимался на чистой машине.

### Что реализовано

**`models/tracker.py` — класс `GranuleTracker`.**

IoU-трекер между соседними кадрами: сравнивает bbox каждой детекции с детекциями предыдущего кадра. Если перекрытие ≥ порога (по умолчанию 0.3) — гранула уже была учтена и не считается повторно. В счётчик и `total_granules` попадают только новые гранулы. Порог настраивается через `PATCH /settings`.

**`api/stream.py` — FPS-based timestamps и frame_skip.**

Для видеофайлов timestamp вычисляется как `frame_index / fps` — привязан к временной шкале видео. Для живой камеры оставлен `monotonic()`. Параметр `frame_skip` позволяет обрабатывать каждый N-й кадр, снижая нагрузку без потери точности.

**`server/requirements.txt` — фикс bcrypt.**

Зафиксирована версия `bcrypt==4.2.1`, совместимая с `passlib 1.7.4`.

## Поправки: Передача размеров источника клиенту
**Дата:** 15.05.2026  
**Статус:** Выполнено

### Проблемы, которые были выявлены

Клиентское приложение отображало bounding box поверх видео, но не могло корректно масштабировать координаты детекций на экран: клиент не знал исходное разрешение видеофайла, переданного на сервер. Координаты bbox приходят в пикселях исходного кадра, и без размеров источника клиент не мог пересчитать их в экранные координаты.

### Что реализовано

В ответ каждого кадра добавлены два поля: `source_width` и `source_height` — разрешение видеоисточника в пикселях. Значения снимаются один раз при открытии файла через `cv2.CAP_PROP_FRAME_WIDTH` / `cv2.CAP_PROP_FRAME_HEIGHT` и передаются вместе с каждым кадром.

Клиент использует эти значения для вычисления масштаба и смещения при отрисовке оверлея с учётом letterbox-отступов.

### Файлы

| Файл | Описание |
|------|----------|
| `api/stream.py` | Чтение `CAP_PROP_FRAME_WIDTH` / `CAP_PROP_FRAME_HEIGHT`, передача в payload |
| `schemas/schemas.py` | Поля `source_width: int` и `source_height: int` в `FrameAnalysisResponse` |
