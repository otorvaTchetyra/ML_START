## Поправки: Трекинг гранул, точность подсчёта и настройки обработки
**Дата:** 11.05.2026  
**Статус:** Выполнено

### Проблемы, которые были выявлены

1. **Двойной счёт гранул.** Одна и та же гранула, видимая в нескольких кадрах подряд, засчитывалась в каждом кадре заново. `total_granules` был завышен.
2. **Некорректные timestamps.** Время кадра бралось через `monotonic()` — реальное время CPU, а не время в видео. При медленном инференсе скользящее окно и интенсивность считались неверно.
3. **Избыточная нагрузка.** YOLOv8 запускался на каждом кадре, хотя гранулы между кадрами почти не меняются.
4. **Несовместимость зависимостей.** `passlib 1.7.4` несовместима с `bcrypt 5.x` — сервер не поднимался на чистой машине.

### Что реализовано

**`models/tracker.py` — класс `GranuleTracker`.**

IoU-трекер между соседними кадрами: сравнивает bbox каждой детекции с детекциями предыдущего кадра. Если перекрытие ≥ порога (по умолчанию 0.3) — гранула уже была учтена и не считается повторно. В счётчик и `total_granules` попадают только новые гранулы. Порог настраивается через `PATCH /settings`.

**`api/stream.py` — FPS-based timestamps и frame_skip.**

Для видеофайлов timestamp вычисляется как `frame_index / fps` — привязан к временной шкале видео. Для живой камеры оставлен `monotonic()`. Параметр `frame_skip` позволяет обрабатывать каждый N-й кадр, снижая нагрузку без потери точности.

**`server/requirements.txt` — фикс bcrypt.**

Зафиксирована версия `bcrypt==4.2.1`, совместимая с `passlib 1.7.4`.

## Поправки: Передача размеров источника клиенту
**Дата:** 15.05.2026  
**Статус:** Выполнено

### Проблемы, которые были выявлены

Клиентское приложение отображало bounding box поверх видео, но не могло корректно масштабировать координаты детекций на экран: клиент не знал исходное разрешение видеофайла, переданного на сервер. Координаты bbox приходят в пикселях исходного кадра, и без размеров источника клиент не мог пересчитать их в экранные координаты.

### Что реализовано

В ответ каждого кадра добавлены два поля: `source_width` и `source_height` — разрешение видеоисточника в пикселях. Значения снимаются один раз при открытии файла через `cv2.CAP_PROP_FRAME_WIDTH` / `cv2.CAP_PROP_FRAME_HEIGHT` и передаются вместе с каждым кадром.

Клиент использует эти значения для вычисления масштаба и смещения при отрисовке оверлея с учётом letterbox-отступов.

### Файлы

| Файл | Описание |
|------|----------|
| `api/stream.py` | Чтение `CAP_PROP_FRAME_WIDTH` / `CAP_PROP_FRAME_HEIGHT`, передача в payload |
| `schemas/schemas.py` | Поля `source_width: int` и `source_height: int` в `FrameAnalysisResponse` |

## Поправки: Исправление клиентского приложения
**Дата:** 15.05.2026  
**Статус:** Выполнено

### Проблемы, которые были выявлены

1. **Вылет при открытии видео.** Клиент использовал библиотеку OpenCvSharp для захвата первого кадра видеофайла. На Linux нативная библиотека `OpenCvSharpExtern` не загружается — приложение падало с `DllNotFoundException` сразу после выбора файла.
2. **Аннотации не отображались.** Подход с рендерингом аннотированного кадра через OpenCV на стороне клиента не работал: после удаления OpenCvSharp замена bounding box на экране отсутствовала полностью.
3. **Ошибка авторизации.** Сервер ожидал JSON с полями `username` и `password` в нижнем регистре, а клиент отправлял `Username` и `Password` с заглавной буквы из-за отсутствия атрибутов сериализации.
4. **Ошибка парсинга URI на Linux.** `Uri.TryCreate("/auth/login", UriKind.Absolute)` на Linux возвращал `file:///auth/login` вместо ожидаемого пути. Запросы авторизации уходили по неверному адресу.

### Что реализовано

**Удаление OpenCvSharp из клиента. Переход на кроссплатформенное решение.**

Пакеты `OpenCvSharp4` и `OpenCvSharpExtern` убраны из проекта. Вся логика захвата и аннотации кадров через `VideoCapture` и `Mat` удалена из `MainViewModel` и `CameraCaptureService`. Размеры видеоисточника теперь берутся из полей `source_width` / `source_height`, которые сервер передаёт в каждом кадре.

Прежний подход работал только на Windows (`OpenCvSharp4.runtime.win`): на Linux нативная библиотека не загружалась. Новый стек полностью кроссплатформенный: воспроизведение видео обеспечивает LibVLC (работает на Windows и Linux), детекция выполняется на сервере, оверлей реализован средствами Avalonia без нативных зависимостей. Коллеги на Windows получают тот же функционал без каких-либо изменений на своей стороне.

**Оверлей bounding box через ItemsControl + Canvas.**

Вместо подмены кадра аннотированным изображением реализован векторный оверлей: `ItemsControl` с `Grid` в качестве панели элементов и нулевым `Canvas` внутри каждого шаблона. `Canvas.Left` / `Canvas.Top` устанавливаются напрямую на `Border`, который является прямым дочерним элементом `Canvas` — это корректный способ позиционирования в Avalonia. Координаты пересчитываются с учётом letterbox-отступов и масштаба при каждом изменении размера области просмотра.

**Исправление авторизации.**

На модель `LoginRequest` добавлены атрибуты `[JsonPropertyName("username")]` / `[JsonPropertyName("password")]`. В `ApiClient.PostAsync` переход на `JsonContent.Create(data, data.GetType())` — это гарантирует применение атрибутов сериализации во время выполнения.

**Исправление URI на Linux.**

В `ApiClient.BuildUri` добавлена проверка схемы: если `Uri.TryCreate` возвращает URI со схемой, отличной от `http` / `https`, он игнорируется и адрес собирается из базового URL. Это исключает подмену на `file://` при обработке относительных путей.

## Поправки: Software rendering оверлея и исправление настроек
**Дата:** 15.05.2026  
**Статус:** Выполнено

### Проблемы, которые были выявлены

1. **Боксы детекций не отображались.** Оверлей на Canvas добавлялся корректно (логи подтверждали наличие прямоугольников), но LibVLC создаёт нативное X11-окно поверх Avalonia — так называемая проблема airspace. Нативное окно перекрывает все управляемые элементы Avalonia, в том числе Canvas с боксами.
2. **Настройки не сохранялись.** `ConfigurationService` использовал относительный путь `"appsettings.json"`. CWD процесса при запуске через `dotnet run` — родительская директория решения, а не директория проекта. `File.Exists("appsettings.json")` возвращал `false`, запись в файл не производилась.
3. **При переходе в настройки и обратно видео пропадало.** `NavigateToMainAsync()` каждый раз создавал новый экземпляр `MainViewModel` (Transient), старый с установленным `VideoPath` выбрасывался.
4. **Ошибка при сохранении настроек.** Файл сохранялся успешно, но последующий вызов `_apiClient.PatchAsync` выбрасывал исключение — весь блок `try` завершался с ошибкой, и пользователь видел сообщение об ошибке вместо подтверждения сохранения.
5. **Некорректное значение IoU в файле.** В `appsettings.json` поле `"Iou"` имело значение `"0,45"` с запятой (русская локаль). `float.Parse` с `CultureInfo.InvariantCulture` не мог распарсить значение.

### Что реализовано

**Переход на software rendering в LibVLC.**

Вместо `VideoView` (нативное окно) теперь используется `Image` с `WriteableBitmap`. LibVLC декодирует кадры в буфер памяти через `SetVideoFormatCallbacks` / `SetVideoCallbacks` (формат BGRA), буфер копируется в `WriteableBitmap` на UI-потоке. Canvas с детекциями лежит поверх `Image` как обычный Avalonia-элемент — проблема airspace устранена.

**Исправление пути к файлу настроек.**

В `ConfigurationService` путь к файлу заменён на `Path.Combine(AppContext.BaseDirectory, "appsettings.json")` — всегда указывает на директорию бинарника, где файл гарантированно присутствует. Аналогичный `SetBasePath(AppContext.BaseDirectory)` добавлен в `ConfigurationBuilder` при старте приложения.

**Возврат к видео без пересоздания ViewModel.**

В `SettingsViewModel.GoToMainAsync` вместо `NavigateToMainAsync()` (создаёт новый VM) теперь вызывается `HostScreen.Router.NavigateBack.Execute()` — навигация возвращается к уже существующему `MainViewModel` на стеке. В `MainView.BindViewModel()` добавлен вызов `PlayPath(vm.VideoPath)` при привязке к VM с уже установленным путём — видео возобновляется после пересоздания View.

**Разделение сохранения файла и вызова API.**

Запись в файл вынесена до вызова `PatchAsync`. Успех сохранения показывается сразу после записи файла. Если сервер недоступен, выводится предупреждение "Настройки сохранены (сервер недоступен)" — локальные настройки при этом уже записаны.

**Исправление значения IoU.**

В `appsettings.json` исправлено `"0,45"` → `"0.45"`.

| Файл | Описание |
|------|----------|
| `Client/Views/MainView.axaml` | Заменён `media:VideoView` на `Image x:Name="VideoImage"` |
| `Client/Views/MainView.axaml.cs` | Software rendering: `VideoFormatCallback`, `LockCallback`, `DisplayCallback`; `PlayPath`; `BindViewModel` с восстановлением видео |

## Поправки: Стабилизация и порог уверенности детекций
**Дата:** 15.05.2026  
**Статус:** Выполнено

### Проблемы, которые были выявлены

1. **Вылет при возврате из настроек.** В `MainView` был обработчик `DetachedFromVisualTree`, который при уходе на экран настроек уничтожал LibVLC и `MediaPlayer`. При возврате View пересоздавался и обращался к уже освобождённым объектам — приложение падало.
2. **Настройки не применялись.** Из-за неверного пути к файлу конфигурации любые изменения порогов в интерфейсе не давали эффекта — детектор всегда работал с захардкоженными значениями.
3. **Дублирующиеся боксы на одном объекте.** Значение IoU по умолчанию `0.45` было ниже стандартного — модель хуже подавляла перекрывающиеся боксы и могла выдавать несколько детекций на один объект.

### Что реализовано

**Устранение вылета.**

Обработчик `DetachedFromVisualTree` удалён из `MainView`. LibVLC инициализируется один раз при создании View и живёт на протяжении всей сессии. Навигация между экранами больше не освобождает медиаплеер.

**Исправление применения настроек детектора и значения IoU.**

До исправления `ConfigurationService` не мог найти файл `appsettings.json` из-за неверного пути — настройки из UI никогда не сохранялись и не применялись. Детектор всегда работал с захардкоженными значениями, независимо от того, что было выставлено в интерфейсе.

После исправления пути настройки сохраняются корректно и применяются при запуске. Теперь значения порогов из UI передаются в модель. Значение IoU по умолчанию поднято с `0.45` до `0.5` — стандартное значение для YOLO, при котором модель корректно подавляет перекрывающиеся боксы.

**Устранение вылета при окончании видео.**

При завершении воспроизведения LibVLC вызывает `VideoCleanupCallback` для освобождения буфера кадра. Одновременно `DisplayCallback` мог держать ссылку на этот же буфер и пытаться прочитать его после освобождения — возникала гонка, приводившая к `munmap_chunk(): invalid pointer` и аварийному завершению процесса.

Доступ к буферу защищён `lock` во всех трёх колбэках (`LockCallback`, `DisplayCallback`, `VideoCleanupCallback`). `DisplayCallback` снимает копию буфера под блокировкой, после чего работает с копией — `VideoCleanupCallback` не может освободить буфер, пока идёт копирование.

### Файлы

| Файл | Описание |
|------|----------|
| `Client/Views/MainView.axaml.cs` | Удалён обработчик `DetachedFromVisualTree`; добавлен `lock` для защиты буфера кадра |
| `Client/appsettings.json` | Исправлено значение `Iou` (запятая → точка); значение по умолчанию поднято с `0.45` до `0.5` |

## Оверлей загрузки с прогрессом
**Дата:** 15.05.2026  
**Статус:** Выполнено

### Проблема

При нажатии кнопки воспроизведения видео загружалось на сервер и анализировалось без какой-либо обратной связи в интерфейсе. Пользователь не понимал, идёт ли процесс или приложение зависло.

### Что реализовано

Поверх видеоэлемента добавлен полупрозрачный оверлей с двумя этапами:

**Этап 1 — загрузка файла на сервер.** `ApiClient.PostFileAsync` оборачивает файловый поток в `ProgressStream`, который уже присутствовал в проекте. По мере отправки байтов прогресс передаётся через `IProgress<double>` в `MainViewModel`. Отображается текст "Загрузка видео на сервер..." и числовой процент.

**Этап 2 — анализ видео.** После завершения загрузки сервер начинает стримить результаты детекции. Оверлей переключается в режим неопределённого прогресса (анимированная полоса) с текстом "Анализ видео...". При получении первого кадра оверлей скрывается и начинается обычное отображение боксов.

### Файлы

| Файл | Описание |
|------|----------|
| `Client/Services/IApiClient.cs` | Добавлен параметр `IProgress<double>? progress` в `PostFileAsync` |
| `Client/Services/ApiClient.cs` | `PostFileAsync` оборачивает поток в `ProgressStream` при наличии `progress` |
| `Client/Services/StreamService.cs` | `SendVideoAsync` принимает `uploadProgress` и `onUploadComplete` |

## Звуковой сигнал и баннер при превышении порога. Исправление вылетов LibVLC
**Дата:** 15.05.2026  
**Статус:** Выполнено

### Что реализовано

**Визуальный баннер при алерте.** При получении кадра с `Threshold_exceeded = true` или `Out_of_schedule = true` поверх видео появляется красный полупрозрачный баннер с текстом ("Превышен порог гранул", "Кормление вне расписания" или оба сразу). Баннер автоматически скрывается через 5 секунд. Повторные алерты срабатывают не чаще одного раза в 5 секунд.

**Звуковой сигнал.** При алерте запускается воспроизведение системного звука через `cvlc --play-and-exit`. На Linux используется `/usr/share/sounds/freedesktop/stereo/alarm-clock-elapsed.oga`, на других платформах — `Console.Beep`.

**Исправление вылетов LibVLC (SIGSEGV / munmap_chunk).** Приложение периодически падало с `munmap_chunk(): invalid pointer` (SIGABRT, exit 134) и SIGSEGV (exit 139). Причина: буфер кадра был выделен через `GCHandle.Alloc(Pinned)` на управляемом массиве. LibVLC обращался к этой памяти из нативного потока, что конфликтовало с .NET runtime при GC или при переключении медиа. Буфер переведён на неуправляемую память через `Marshal.AllocHGlobal` — LibVLC всегда работает со стабильным нативным указателем вне досягаемости GC. Все операции с буфером защищены `lock`.

| Файл | Описание |
|------|----------|
| `Client/ViewModels/MainViewModel.cs` | Свойства `IsAlertActive`, `AlertText`; запуск звука и баннера при алерте; `PlayAlertSoundAsync` через `cvlc` |
| `Client/Views/MainView.axaml` | Красный баннер поверх видео, привязан к `IsAlertActive` и `AlertText` |
| `Client/Views/MainView.axaml.cs` | Буфер кадра переведён с `GCHandle.Pinned` на `Marshal.AllocHGlobal`; все колбэки под `lock` |

## Отображение гранул и интенсивности. Фильтр детекций по размеру. Исправление ошибок воспроизведения
**Дата:** 15.05.2026  
**Статус:** Выполнено

### Что реализовано

**Панель гранул и интенсивности.** В левом нижнем углу видео добавлен полупрозрачный HUD с двумя показателями: текущее число гранул (`Granule_count`) и интенсивность подачи корма (`Intensity_per_sec`, гр/с). Данные обновляются при каждом кадре. Свойства `GranuleCount` и `IntensityPerSec` добавлены в `MainViewModel`.

**Фильтр детекций по размеру бокса.** В `GranuleDetector.detect` на сервере добавлена проверка: если ширина или высота бокса превышает 12% от соответствующего размера кадра — детекция отбрасывается. Гранулы корма мелкие, крупные объекты (рыбы) таким образом отсеиваются без переобучения модели.

**Исправление чёрного экрана при запуске.** `_mediaPlayer.Stop()` внутри `PlayPath` вызывал событие `EndReached`, которое срабатывало как "конец видео" и останавливало анализ до его начала. Добавлен флаг `_ignoreEndReached` — пока идёт намеренная остановка перед запуском нового видео, событие игнорируется.

**Непрозрачный загрузочный оверлей.** Фон оверлея изменён с `#CC000000` (80%) на `#FF000000` (100%) — старые боксы детекции больше не просвечивают сквозь него во время загрузки нового видео.

**Автоостановка при окончании видео.** При срабатывании `EndReached` (видео дошло до конца) экран очищается и анализ останавливается автоматически — боксы детекции больше не висят на чёрном экране.

## Безопасность: rate limiting и смена JWT-ключа. Фикс утечки сессии при выходе
**Дата:** 15.05.2026  
**Статус:** Выполнено

### Что реализовано

**Rate limiting на endpoint входа.** С помощью библиотеки `slowapi` добавлено ограничение: не более 5 запросов к `POST /auth/login` в минуту с одного IP. При превышении сервер возвращает `HTTP 429 Too Many Requests` с заголовком `Retry-After`.

**Отображение попыток и таймера на клиенте.** После каждой неудачной попытки входа показывается оранжевое предупреждение «Осталось попыток: X». При получении 429 от сервера — красный баннер с обратным отсчётом «Повторите через Xs», кнопка «Войти» блокируется до окончания таймера.

**Замена JWT secret key.** На сервере в файле `.env` заменён слабый ключ `super-secret-key-dronevision-2026` на случайный 64-символьный hex-ключ. Файл `.env` не попадает в git (`.gitignore`). Все ранее выданные токены автоматически инвалидировались.

**Фикс утечки сессии при выходе.** При нажатии «Выйти» в `MainViewModel` не вызывался `StopVideoAndAnalysisAsync` — `StreamService` продолжал читать поток и вызывать колбэк `OnStreamFrame` со старого экземпляра VM. Это приводило к тому, что алерт срабатывал на экране авторизации и в новой сессии другого пользователя. Исправлено: перед logout и перед навигацией в «Настройки»/«Стрим» теперь принудительно вызывается остановка анализа и сброс `IsAlertActive`.

## Стабилизация воспроизведения. Настройка порога гранул. Дроссель событий на сервере
**Дата:** 15.05.2026  
**Статус:** Выполнено

### Проблемы, которые были выявлены

1. **Видео зависало в середине.** Сервер сохранял `FeedingEvent` в БД на каждом кадре с превышением порога (~8 раз в секунду). Одновременно `RecordFrameAlertAsync` вызывал `LoadJournalAsync` после каждого алерта — UI-поток перегружался запросами к серверу. Видео замирало, к кадру ~4700 процесс зависал.
2. **Чёрный экран после повторного запуска.** `_ignoreEndReached` был обычным полем — изменение в UI-потоке не было видно VLC-потоку. Spurious `EndReached` после `Stop()` останавливал анализ до начала нового видео.
3. **Преждевременное "Анализ остановлен".** При естественном окончании видео показывалось то же сообщение, что и при ручной остановке.
4. **Кнопка настроек долго открывалась.** `InitializeAsync` ждал `GET /settings` перед навигацией — сеть блокировала переход.

### Что реализовано

**Дроссель событий на сервере.**

В `api/stream.py` добавлен таймер: между записями в `FeedingEvent` должно пройти не менее 30 секунд. Это снижает нагрузку на БД с ~8 записей/сек до 2 записей/мин при непрерывном превышении порога.

**Предвыделенный управляемый буфер в DisplayCallback.**

`byte[] _managedBuffer` выделяется один раз в `VideoFormatCallback`. В `DisplayCallback` вместо `new byte[]` на каждый кадр используется уже существующий массив — нет давления на GC.

**volatile `_ignoreEndReached`.**

Поле помечено `volatile` — изменение в UI-потоке немедленно видно VLC-потоку без барьеров памяти. `_ignoreEndReached = true` устанавливается перед каждым `_mediaPlayer.Stop()`.

**PlayPath как async Task.**

`PlayPath` переведён из `async void` в `async Task`. `RestartStoppedPlayerAsync` теперь корректно awaits его, и `_ignoreEndReached = false` устанавливается только после завершения 300 мс задержки.

**Разделение "Видео завершено" и "Анализ остановлен".**

`StopVideoAndAnalysisAsync` получил параметр `videoEnded`. При естественном конце видео показывается "Видео завершено", при ручной остановке — статус остаётся пустым.

**Загрузка серверных настроек fire-and-forget.**

`LoadServerSettingsAsync()` вызывается без `await` в `LoadData` — навигация в «Настройки» не блокируется сетевым запросом.

**Настройка порога гранул в UI.**

В экране «Настройки» добавлен слайдер «Порог гранул» (диапазон 1–200). Значение загружается с сервера при открытии страницы и сохраняется через `PATCH /settings`.

**Удаление настройки Frame Skip.**

Слайдер Frame Skip убран из интерфейса. Измерения показали, что при максимальном значении 4 параметр давал прирост производительности не более 2% при заметном ухудшении точности подсчёта — соотношение неприемлемое.
